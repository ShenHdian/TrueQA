# TrueQA 部署流程参考

> 目的：回答「为什么需要服务器 / 为什么需要 HTTPS 域名 / 正确部署流程是什么」三个问题，作为当前方案的合理性评估依据。

---

## 1. 为什么还需要服务器？

小程序客户端本身跑在微信里（代码通过 `miniprogram-ci` 上传到微信），**但它不是孤岛**——下面几件事必须有一个持续运行的服务端，且这部分逻辑**不能放进小程序前端**：

| 必须由服务端承担的职责 | 原因 |
| --- | --- |
| **登录换 openid** | `wx.login()` 拿到的 `code` 必须调用微信 `code2Session` 接口（带 **AppSecret**）才能换到用户身份。AppSecret 绝不能写进小程序前端（会被反编译窃取），所以必须在服务端持有。 |
| **数据持久化** | 用户自建题、怒气值、对战记录、等级/经验等，需要**跨设备、跨会话、跨用户**共享。小程序的 `wx.setStorageSync` 只是**本机本地缓存**，换手机/清缓存就丢，也不能被对手读到。 |
| **实时对战（P3）** | 2 人西部对战需要 WebSocket 服务端做房间状态同步、回合广播——这是一个**常驻进程**，不是「被调用一次」就能解决的。 |
| **AI 评分（后续）** | 调外部大模型需要 API Key，Key 同样只能放服务端。 |

**结论**：服务器不是「为了存数据才需要」这么简单，它是**密钥保险箱 + 共享数据库 + 实时管道**三合一。所以数据保存在服务器上只是原因之一，且是核心原因。

### 关于「服务器端跑服务」还是「GitHub Actions 自动同步部署」

这是把**两件事**混在了一起，它们是不同层面的：

- **运行时（Runtime）**：一个**常驻进程**，7×24 在服务器上监听请求（`pm2` / `systemd` 拉起 `node src/index.js`）。它不是「被触发」的，是持续活着的。
- **部署（Deploy）**：把代码/更新**送到**服务器的方式。这部分**确实可以用 GitHub Actions 自动化**——push 到 main 时，Actions 通过 SSH 登录服务器、拉最新代码、重启服务。

所以正确表述是：**两者都要**——
> 服务器上跑一个常驻服务（pm2）；同时用 GitHub Actions 工作流在每次 push 时自动把新代码同步上去并重启。

而不是「二选一」。纯靠 Actions 在被触发时跑一下、不保留常驻进程，是无法对外提供实时 API 的。

> 替代方案（你之前选了自建 Node，这里仅作对照）：**微信云开发（CloudBase）** 把数据库/云函数/存储/WebSocket 全托管了，**不需要自购服务器、不需要域名备案**，但受微信云环境额度与生态约束。若不想运维，可考虑后期迁移。

## 2. 为什么一定要 HTTPS 域名？不能直接用服务器 IP 吗？

这是微信平台的**硬性约束**，不是我们偷懒：

### 2.1 硬规则（来自微信小程序平台要求）
1. **合法域名白名单只接受域名，不接受 IP**。`wx.request` 只能访问在 MP 后台「服务器域名 → request 合法域名」里登记的地址，而**该字段只认域名，填 IP 会被拒**。所以 `http://1.2.3.4:3000` 根本登记不进去。
2. **生产环境强制 HTTPS**。正式版/体验版里所有 `wx.request` 必须是 `https://`；明文 `http://` 会被微信直接拦截（只有 DevTools 勾了「不校验合法域名」、或本地调试才放行）。
3. **WebSocket 同理要 `wss://` + 已登记 socket 域名**。P3 对战的实时通道也必须走加密域名。
4. **域名需 ICP 备案**（服务器在大陆时）。微信用备案证明你对域名的控制权与合规身份。

### 2.2 域名在这套架构里的具体作用
- **信任与合规锚点**：备案 + 有效证书 = 微信认可「这个后端是真实可控的」，这是平台治理手段。
- **稳定寻址**：域名映射到后端 IP，以后你换服务器/做负载均衡/换机房，**客户端代码一行都不用改**；IP 一旦变动，所有用户端全挂。
- **端口收敛**：白名单域名不带端口，统一走 443；用 IP 往往要带 `:3000` 等非标端口，配置更乱、且无法在白名单体现。
- **归属清晰**：云服务器 IP 可能随实例重生而变化，域名是你的、不变。

### 2.3 那 IP 到底能不能用？
- **DevTools 模拟器**（勾「不校验合法域名」）：可以直连 `127.0.0.1:3000` 或局域网 IP——这正是你现在本地能跑通的原因。
- **真机 / 体验版 / 正式版**：**不行**。CI 生成的预览二维码扫出来是体验版，必须走 HTTPS 域名。所以你「想直接在真机用」的那一步，卡点就在域名+备案上。

## 3. 完整部署流程参考

下面是一套可落地的正确流程（当前项目已具备其中「小程序发布」一半，后端部署部分待你执行）。

### 阶段 A：一次性基础设施（你侧，不可逆返工）
1. **买域名**并 **ICP 备案**（若服务器在大陆；境外服务器可免备案但国内访问慢且部分场景受限）。
2. **申请 HTTPS 证书**：Let's Encrypt 免费证书，或由云厂商一键签发。
3. **云服务器**：装好 Node（≥18），准备用 `pm2` 守护 `server/src/index.js`。
4. **微信公众平台**：拿到真实 AppID（替换 `mp/project.config.json` 的 `touristappid`）；「开发设置 → 小程序代码上传密钥」生成并下载 `private.key`（勾「不限制 IP」）。
5. **MP 后台 → 开发管理 → 服务器域名**：登记
   - request 合法域名：`https://你的域名`
   - socket 合法域名：`wss://你的域名`
   - （upload/download 按需）
6. **GitHub Secrets**：`WX_APPID`、`WX_PRIVATE_KEY`（pem 全文）、以及后端部署用的 `DEPLOY_HOST` / `DEPLOY_USER` / `DEPLOY_SSH_KEY`。

### 阶段 B：后端部署（常驻服务 + 自动同步）
**方式一（推荐，自动化）**：在现有 Actions 工作流里加一个 `deploy` job，push 到 main 时：
```
checkout → 通过 SSH 登录云服务器 → git pull / 拷贝 server/ → pm2 reload → 完成
```
需把服务器 SSH 私钥作为 Secret，且服务器预先装好 Node + pm2。

**方式二（手动）**：自己 SSH 上服务器，`git pull` + `pm2 restart`。

无论哪种，**Nginx 反向代理**都要配：
```
443 (HTTPS) → Nginx → localhost:3000 (Node)
并且对 /ws 做 WebSocket 升级（proxy_set_header Upgrade / Connection）
```
证书由 Nginx 终结 TLS，Node 只跑内网 `localhost`。

> 生产建议：当前 `server/data/db.json`（JSON 文件）仅适合开发；上线前迁移到 SQLite/Postgres/Redis，避免并发写损坏。

### 阶段 C：小程序发布（已跑通）
```
git push main → GitHub Actions:
  ├─ mini-program job → miniprogram-ci 编译 → 预览二维码(Artifact) / 或上传体验版
  └─ deploy job       → SSH 同步后端（阶段 B）
```
- 预览二维码：下载 Artifact，微信扫一扫真机预览（验证 UI/交互）。
- 体验版满意后：MP 后台「提交审核」→ 微信审核 → 发布正式版。

### 阶段 D：联调校验清单
- [ ] 真机扫码能**抽到系统题**（证明 HTTPS 域名 + 白名单 + Nginx 全通）
- [ ] 加一道用户题后，换设备登录同一账号仍能看到（证明数据在服务器、跨设备）
- [ ] 后端进程重启后 `pm2` 自动拉起（证明常驻）
- [ ] 改一处后端代码 → push → Actions 自动部署生效（证明 CI 同步链路）

## 4. 当前方案合理性评估

| 项 | 现状 | 评价 |
| --- | --- | --- |
| 小程序 CI（编译+预览二维码） | 已跑通 | 合理，标准做法 |
| 后端常驻服务 | 仅本地 `:3000` 跑 | 缺：需上云 + pm2 |
| 后端自动部署 | 未做 | 建议加 SSH deploy job |
| HTTPS 域名 | `PROD.BASE_URL` 仍是占位 | 缺：需买域名+备案+证书 |
| MP 域名白名单 | 未登记 | 缺：上线前必做 |
| 数据存储 | JSON 文件 | 开发 OK，生产需换真 DB |
| 密钥管理 | AppSecret 在服务端 | 合理（未泄露到前端）✅ |

**一句话结论**：架构方向正确（服务端常驻 + 小程序 CI 上传 + 域名白名单是微信生态标准范式）。卡在「后端上云 + HTTPS 域名 + 白名单」这三件你侧的基础设施上，不是代码问题。
