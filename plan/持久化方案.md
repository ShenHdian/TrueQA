# 持久化方案：JSON 加固（当前）→ SQLite（P3 对战时）

> 背景：用户真机上线在即，担心 JSON 文件存储重启/崩溃丢数据。本文给出分阶段方案与决策依据。

## 一、当前存储机制现状

`server/src/store.js` 提供 `read()/write()`，`index.js` 在**模块顶层** `let db = read()` 只在进程启动时把 `db.json` 读进内存一次；之后所有请求改的是内存 `db` 对象，`write(db)` 仅把内存快照落盘。

**结论：db 本质是「内存库 + 定时落盘」，不是每次请求都读盘。**

## 二、真实风险 vs 误判

| 顾虑 | 是否真实风险 | 说明 |
| --- | --- | --- |
| 重启丢数据 | ❌ 否 | `db.json` 在磁盘，pm2 restart 重新 `read()` 读回，不丢 |
| 并发读写竞态 | ❌ 否（当前） | 单进程 + 内存 db，JS 单线程，read-modify-write 不交叉 |
| 写一半崩溃损坏文件 | ✅ **是** | 旧 `write` 用 `writeFileSync` 直接覆盖，进程在写中段被 kill 会留下半截 JSON；下次启动 `read()` 的 catch 返回空默认库 → **数据清空** |
| 部署 scp 覆盖丢失 | ❌ 否 | `server/data/db.json` 已被 `.gitignore` 排除，CI 不会传它，服务器上的生产数据保留 |
| 单实例内存分叉 | ⚠️ 仅未来 | 若 P3 用 cluster/多实例，各进程内存 db 不一致（SQLite 自然解决） |

**唯一致命风险 =「写一半崩溃损坏 db.json」**。

## 三、阶段一（已落地）：加固 JSON 存储 —— 零依赖

改动 `server/src/store.js`：

1. **原子写**：先写 `db.json.tmp`，再 `rename` 覆盖主文件（同卷 rename 原子，不会出现半截文件）。
2. **写前备份**：主文件完好时 `copyFileSync` 到 `db.json.bak`，崩溃最多回退到上一版本。
3. **损坏恢复**：`read()` 时主文件解析失败自动回退备份；备份也坏则初始化默认并打印告警（不静默丢数据）。
4. **保持同步 API**：`write(db)` 仍是同步函数，`index.js` 零改动；不引入任何 npm 依赖，守住「`node src/index.js` 直接跑」的便利性。

> 已提交。验证：重启后端 health OK、抽题/加题/改怒气值全链路正常，损坏恢复逻辑经代码审查成立。

## 四、阶段二（P3 西部对战再做）：上 SQLite

**何时做**：当要做 P3 实时对战（多用户高频改血量/等级），或预期用户量上来需要多实例部署时。

**为什么不在现在做**：引入 SQLite 需 `npm install better-sqlite3`（原生模块，要编译工具链），破坏「零依赖直接跑」；当前数据量（每用户几道题）JSON 完全够用，过度工程。

**技术选型**（届时二选一）：

| 方案 | 优点 | 代价 |
| --- | --- | --- |
| `better-sqlite3` | 同步 API、快、事务支持好 | 原生编译，需 `npm install` + 构建工具，CI 需装 build deps |
| `sql.js`（WASM） | 纯 JS、无原生编译 | 仍是一个 npm 依赖；大文件需自行管理持久化 |

**改造清单（届时）**：
1. `store.js` 重写为 SQLite 封装（open / getDb / 建表 / 迁移）。
2. 数据模型：users、questions、battle_rooms、battle_rounds 等表。
3. seed：`seed.js` 改为首次建表时插入 10 道系统题（或 `INSERT OR IGNORE`）。
4. 迁移：`db.json` 现有数据写个一次性迁移脚本导入 SQLite（用户题、怒气值、等级）。
5. 部署：服务器 `npm install` 后 `pm2 restart`；`.gitignore` 追加 `*.sqlite`。
6. 多实例：SQLite 单文件可多进程读，但写需 WAL 模式；真要高并发建议换 MySQL/Postgres。

## 五、推荐决策

- **现在**：用阶段一加固 JSON，零依赖、立即可用、消除唯一致命风险。
- **P3 启动时**：再上 SQLite（推荐 `better-sqlite3`），并借机把对战实时状态从内存移入数据库。

> 不要现在为了「看起来专业」引入 SQLite——它带来的原生编译负担会反噬你最看重的本地直跑与 CI 简洁性，而当前 JSON 在加固后已足够稳健。
